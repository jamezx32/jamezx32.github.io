在 64 位平台上，如果仍然直接用 64 位值去保存堆对象引用，那么 V8 在对象字段、元素槽、属性槽里存放大量 JavaScript 值时，内存开销会比较明显。指针压缩就是为了解决这个问题。它的做法不是简单把 64 位地址“砍半”，而是在启用该机制的 64 位构建里，把很多 tagged value 以更紧凑的 32 位形式存进内存，真正使用时再恢复成可参与运算的 64 位形式。

这里默认讨论的是启用了 Pointer Compression 的 64 位 V8。V8 官方在 2020 年的文章里提到，真实网页场景中 tagged value 大约能占整个 V8 堆的 70%。这个数字更适合拿来说明压缩收益很大，不适合当成所有版本和所有环境下都固定成立的常量。

## tagged value

在 V8 里，JavaScript 的值并不是一律按普通 64 位裸指针处理。很多值都会采用 tagged value 的形式编码，粗略看主要就是两类：一种是直接编码在值里的 `Smi`，另一种是指向堆对象的 `HeapObject` 引用。调试时最低位通常很有参考价值，常见情况下 `0` 更接近 `Smi`，`1` 更接近 heap object 指针类值，但这说的是 tagged value 的表示语义，不是说任意 C++ 指针字段都能直接这样看。再往下看时，还要把 strong / weak 之类的 tag 语义一起带上，不能只盯着最低位。

## 为什么要压缩

指针压缩依赖的前提是，让相关堆对象落在一个可以用 32 位偏移描述的 4GB 地址范围里。这样对象引用在内存里就不必总是保存完整的 64 位地址，而是可以保存相对某个 base 的压缩表示。真正被压缩的，是大量以 tagged 形式存放的堆对象引用的内存表示，而不是说 V8 里所有指针从此都只有 32 位。

## 压缩后的表示

如果只从“内存里怎么存”这个角度看，压缩后的 tagged value 可以先粗略理解成下面这样：

```txt
|---------- 32 bits ----------|
Pointer: |____30位有效载荷____w1|
Smi:     |_____31位整数值______0|
```

这里的 `w` 是对象指针上的附加标记位，用来区分 strong pointer 和 weak pointer。`Smi` 仍然直接编码在这 32 位里，而对象引用保存的是可用于恢复地址的压缩表示。由于对象按字对齐分配，真正能拿来表达地址信息的有效位并不是完整 32 位；压缩后的 `Smi` 也更接近 32 位架构下的 `int31` 表示。Pointer Compression 本质上是重组了 tagged value 的表示方式，让它在内存里更紧凑。

## 调试时容易混淆的地方

最容易混淆的一点，是把它理解成“对象地址只剩 32 位”。这个说法并不准确，压缩的是存储表示，不是说运行时永远只拿着 32 位地址做事；读取、解引用和地址计算时，V8 还是会把它恢复成可用的 64 位形式。另一个容易混淆的点是 4GB 这个限制。这里说的是压缩表示依赖的地址范围，不是整个进程只能有 4GB 内存。把这两点分清之后，再去看对象布局、tagged value 和利用里的地址计算，会顺很多。
