在 64 位平台上，如果仍然直接使用 64 位指针来表示堆对象引用，那么 V8 在保存大量 JavaScript 值时会带来比较明显的额外内存开销。

指针压缩（Pointer Compression）的核心思路可以概括成一句话：**不再直接存完整地址，而是只存相对某个基址的 32 位压缩表示；真正使用时，再把它还原成 64 位地址。**

这套机制主要服务于 V8 中占比很高的 **tagged value**。从 V8 官方给出的数据来看，在真实网页场景里，tagged value 大约能占到整个 V8 堆的 70%，所以压缩这里的收益是非常可观的。

---

### 1. 先看 V8 的 tagged value

在 V8 里，JavaScript 的值并不是一律按“普通 64 位裸指针”处理。很多值都会采用 **tagged value** 的形式编码。先粗略理解的话，可以把它们分成两类：

- **Smi**：直接编码在值里的小整数；
- **HeapObject 引用**：指向堆对象的 tagged pointer。

在常见调试语境里，最低位通常可以帮助我们快速判断一个值的大致类别：

- `0`：通常表示 `Smi`
- `1`：通常表示 heap object 指针类值

不过这里还要注意，V8 的 tag bits 不只是用来区分 `Smi` 和对象引用；对于对象指针来说，另一个 tag bit 还会参与区分 strong pointer 和 weak pointer。

所以在调试引擎、看内存布局、或者分析对象表示时，先观察最低位往往是一个很实用的切入点，但不能把它简单理解成“最低位 1 就已经说明了全部语义”。

---

### 2. 为什么要做指针压缩

问题本质上很直接：在 64 位架构下，一个完整指针要占 **8 字节**。而 V8 的对象字段、元素槽、属性槽里又会存放大量 tagged value，这些字段一多，整体内存成本就会被迅速放大。

V8 的优化思路是：

- 让相关对象分配在一个可由 32 位偏移表示的 **4GB 地址范围**内；
- 不再保存完整的 64 位地址；
- 只保存相对于某个 base 的 **32 位压缩表示**。

这样一来，原本需要 64 位保存的堆对象引用，就可以在存储时只占 32 位。对于大量 tagged value 来说，这会显著降低整体内存占用。

---

### 3. 压缩后的基本表示

可以先把压缩后的 tagged value 粗略理解成下面这样：

```txt
(压缩后统一存为 32 位)
|---------- 32 bits ----------|
Pointer: |____30位有效载荷____w1|
Smi:     |_____31位整数值______0|
```
- **Smi** 仍然直接编码在这个 32 位值里；
- **对象引用** 不再直接保存完整地址，而是保存可用于恢复地址的压缩表示。

其中 `w` 是对象指针上的附加标记位，用于区分 strong pointer 和 weak pointer。更准确地说，压缩 pointer 保存的是 4GB 范围内的 offset；压缩 Smi 则更接近 32 位架构下的 int31 表示。因此，pointer compression 并不是把 64 位值“直接砍半”，而是重新组织了 tagged value 的表示方式。
