    V8 堆的内存布局

    V8 使用堆内存来管理 JavaScript 运行时动态分配的对象、数组、函数对象以及其他复杂数据结构。为了兼顾分配效率、垃圾回收开销和执行性能，V8 会把堆划分为多个具有不同职责的空间。理解这些空间的用途，有助于分析对象分配、GC 行为以及一些与引擎内部实现相关的问题。

    1. 新生代（New Space）¶

    新生代主要用于存放生命周期较短的对象。绝大多数新创建的普通对象，都会优先分配到这里，因此这里也是垃圾回收最频繁的区域。

    新生代通常分为两个半区：

    • From Space：当前正在使用的区域。
    • To Space：垃圾回收时用于接收存活对象的备用区域。

    当发生 Minor GC 时，V8 会把仍然存活的对象从 From Space 复制到 To Space，然后交换两个空间的角色。由于大部分新对象往往“创建后很快死亡”，这种复制式回收在新生代中通常非常高效。

    可以把它理解为：

    • 新对象先进入新生代；
    • 很快死亡的对象会在下一次回收中被直接丢弃；
    • 经过多次回收仍然存活的对象，才会被提升到老生代。

    2. 老生代（Old Space）¶

    老生代用于存放生命周期较长的对象。如果一个对象在新生代中经历多次垃圾回收后仍然存活，V8 往往会把它晋升到老生代。

    这类对象通常包括：

    • 长时间存活的业务对象；
    • 全局对象引用的数据；
    • 缓存结构；
    • 被闭包长期捕获的对象。

    由于老生代中的对象不再像新生代那样“快速死亡”，其垃圾回收策略通常采用：

    • 标记-清除（Mark-Sweep）
    • 标记-整理（Mark-Compact）

    这类回收方式相比新生代复制回收更重，但适合处理存活时间更长、分布更稳定的对象。

    3. 大型对象区（Large Object Space, LOS）¶

    大型对象区用于存放体积较大的对象。这类对象如果仍然采用普通复制式回收，开销会很高，因此通常会直接分配到专门的大对象空间中。

    这一空间的特点是：

    • 适合存放尺寸较大的对象；
    • 一般不会像新生代那样频繁搬移；
    • 垃圾回收通常依赖标记类算法，而不是常规复制回收。

    很多资料会用“大于某个阈值”来描述什么对象会进入 LOS，但这个阈值并不适合在笔记里写死，因为它可能随 V8 版本、对象类型和实现细节而变化。更稳妥的理解方式是：当对象大到不适合频繁复制时，V8 往往会把它放进 Large Object Space。

    4. 代码空间（Code Space）¶

    代码空间主要用于存放 V8 生成的机器码，例如：

    • Baseline / JIT 编译后的代码；
    • 内建函数（builtins）的机器码；
    • 与执行优化相关的代码对象。

    这部分空间和普通对象空间不同，因为其中存放的不是一般的 JavaScript 对象数据，而是可执行代码及相关元数据。

    在分析 JIT、内联缓存、优化代码失效或代码对象布局时，Code Space 是一个非常重要的概念。

    5. 只读空间（Read-Only Space）¶

    V8 还包含一些只读或特殊用途的空间，例如 Read-Only Space。这类空间通常用来存放：

    • 不应在运行时被修改的对象；
    • 常量数据；
    • 某些共享的只读结构。

    把这部分数据放到只读空间中，有助于：

    • 减少重复分配；
    • 提高共享效率；
    • 降低被意外修改的风险。

    6. 指针压缩（Pointer Compression）¶

    在 64 位平台上，V8 常使用指针压缩（Pointer Compression）来降低内存占用。

    它的核心思路不是“改变对象属于哪个空间”，而是：

    • 在特定堆布局前提下，
    • 用更紧凑的方式表示堆内引用，
    • 从而减少对象字段和指针存储带来的开销。

    因此，指针压缩更适合被理解为一种对象表示和地址编码优化机制，而不是独立的堆空间。

    7. 对象生命周期与代际关系¶

    从对象生命周期的角度，可以把 V8 堆的大致行为理解为：

    • 生命周期较短的对象通常先分配在新生代；
    • 经过多次 GC 仍然存活的对象会逐步晋升到老生代；
    • 体积特别大的对象可能直接进入大型对象区；
    • JIT 生成的机器码则位于代码空间。

    这也是 V8 采用分代垃圾回收设计的核心原因：

    大多数对象死得很快，少数对象会活得很久。

    围绕这一经验规律，把不同生命周期、不同用途的对象放到不同空间中，能够显著提升整体内存管理效率。

    8. 一个更实用的理解方式¶

    如果只是为了建立整体印象，可以把 V8 的堆布局先粗略理解成下面这张图：

    • New Space：新创建、短命对象的主要活动区域；
    • Old Space：长期存活对象所在区域；
    • Large Object Space：不适合频繁复制的大对象；
    • Code Space：JIT 与内建代码对象；
    • Read-Only Space：只读共享数据；
    • Pointer Compression：对象表示层面的内存优化机制。

    这套划分并不是为了记概念而记概念，而是因为它直接影响：

    • 对象最初会分配到哪里；
    • 垃圾回收如何发生；
    • 对象会不会被搬移；
    • 某些内部指针和对象地址该如何理解；
    • 调试和漏洞分析时，不同对象为什么会出现在不同区域。
